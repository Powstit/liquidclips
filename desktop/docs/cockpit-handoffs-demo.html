Cockpit v7, handoff contract

Four handoffs visualised. The cockpit OWNS the focused clip's layout/audio/offset/schedule/publish. It DELEGATES caption styling, source-picking, channel-routing, and full-editor power moves to surfaces designed for those jobs. While a handoff is open, the conflicting cockpit controls dim. Return path is always onProjectChange(nextProject).

HANDOFF 01 · ⋮ MENU EXPANDED

The ⋮ menu groups handoffs into Per-clip, Project, Vanity, Navigation. Keyboard chords surface next to each row. The streak / V70 / counters that v6 had in the status strip live here as vanity readouts, not action targets.

HANDOFF 03 · SOURCE CHANGE → pickOverlaySource (per-clip source picker)

Click "Change" on the source row. pickOverlaySource dialog opens. Cockpit's tiles + Advanced panel dim to 40%. User picks → dialog resolves with { kind: "pick", path } → applyOverlay fires with new path → cockpit re-reads clip.overlay.source_path. If user cancels: dialog returns { kind: "cancel" } → no write → cockpit re-enables.

HANDOFF 05 · OPEN FULL EDITOR → ClipPreview modal (Reaction tombstone)

From ⋮ → Open full editor (or keyboard ↵). ClipPreview modal opens — it's the canonical full editor with title/desc/trim/etc. Cockpit's Reaction module becomes a tombstone ("Editor open. Reaction lives in the modal."). The cockpit can't write clip.overlay while the modal is open. This is the load-bearing modalOpen prop from v0.7.27.
